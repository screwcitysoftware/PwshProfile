# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Working in this repo

**Start every task in plan mode.** Explore the code and present a plan for approval before making edits — the session defaults to plan mode (`defaultMode` in `.claude/settings.local.json`). For a **trivial change** (a typo, a one-line doc tweak, a rename), proactively tell the user they can bypass the full workflow — accept the plan immediately, or toggle plan mode off — rather than walking the whole plan/explore/design loop for something that doesn't need it.

## What this is

A PowerShell (pwsh) module holding the reusable building blocks of the user's profile. It is imported (via the `.psd1` manifest) at the top of `Microsoft.PowerShell_profile.ps1`, and its exported functions drive profile startup. There is no build step or linter config. The README documents each exported function's behavior — read it before changing a function, and update it when behavior changes.

The name nods to "Screw City", a nickname for Rockford, Illinois (a historic screw/fastener/machine-tool manufacturing hub). Rockford is *also* the "Forest City", and the module ships two bundled oh-my-posh themes that lean into that dual identity: `screwcity` (purple/blue, signature `#c9aaff`, 🔩 `:nut_and_bolt:` step icon, "Screw City" display name) and `forestcity` (green/brown/gray, signature `#8fce72`, 🌳 `:deciduous_tree:` step icon, "Forest City" display name). Each theme carries a matching **display name, banner color, step icon, fd `LsColors`, and fzf `FzfColors`**, resolved through `Private/Prompt/Get-BundledThemeBranding.ps1`; a custom theme falls back to the neutral `screwcity` branding. The banner **text** is *not* theme-specific — it defaults uniformly to `$env:COMPUTERNAME` (the machine name) for every theme, custom included.

## Development

Reload after changes:

```powershell
Import-Module ./ScrewCitySoftware.PwshProfile.psd1 -Force
```

Run the tests (Pester 5; `Install-PSResource Pester` if missing):

```powershell
Invoke-Pester -Path ./Tests
```

For console-rendering changes (`Invoke-Step`), also eyeball the output in a fresh `pwsh -NoProfile` session with the module imported (e.g. `Invoke-Step "x" { Start-Sleep -Milliseconds 50 }`) — the Spectre spinner only renders in an interactive console, so tests can't see it (they mock the Spectre cmdlets instead).

**After every code change, check `README.md` for documentation that needs updating** — it's the source of truth users read via `Show-PwshProfileReadme`, and it drifts easily. Review (and update where the change touches them): the affected function's `### ` section, parameter bullets, and examples; the **Layout** file tree (`Public/`/`Private/` listings) when files are added/removed/renamed; the `FunctionsToExport` list in the `.psd1`; the Installation/Usage walkthroughs; and any cross-references or counts (e.g. the bundled-font count). The module-level test only enforces that each exported function has *a* README section plus comment-based help — it does **not** verify the prose is current, so keeping it accurate is on you. Also update the function's comment-based help in the same change.

## Architecture

**Standard Public/Private layout, grouped into feature subfolders.** One exported function per file under `Public/`, file named exactly after the function (`Verb-Noun.ps1`). Both `Public/` and `Private/` are organized into parallel feature subfolders — `Install/`, `Startup/`, `Prompt/`, `Tools/` (tool *and* completion enablers), `Rendering/`, `Docs/`, `Core/` — and a function lives in the subfolder matching its responsibility. The `.psm1` is a generic loader (recursively dot-sources `Public/**/*.ps1` and `Private/**/*.ps1` if present, exports public base names) — it never needs editing. **Subfolder nesting is purely organizational: it never affects which functions are exported.** The manifest carries an explicit *flat* `FunctionsToExport` list, and the module-level test globs `Public/` recursively. Adding a function requires three touches: the new `Public/<area>/Verb-Noun.ps1`, its name in `FunctionsToExport` in the `.psd1`, and a README section. Internal helpers go in the matching `Private/<area>/` subfolder. A module-level test enforces that Public file names (across all subfolders), manifest exports, and actual exports all match, and that every function has comment-based help.

**`Invoke-Step` nesting protocol** — the invariant everything else renders through. `Public/Rendering/Invoke-Step.ps1` is the dispatcher (module-scoped state is declared at the top of that file, outside the function, initialized once at import); the per-step worker lives in `Private/Rendering/Invoke-StepInternal.ps1`.
- The invariant: the top-level `Invoke-Step` call opens the transient status spinner (`Invoke-SpectreCommandWithStatus`) and stashes its `[Spectre.Console.StatusContext]` in `$script:StepStatusContext`; nested calls see the context and only push/pop the `$script:StepPath` breadcrumb stack, updating `$ctx.Status` to the joined path (prefixed by `$script:StepRootIcon`, the top-level step's icon). The spinner clears itself when the top-level scriptblock returns, and the dispatcher then writes one permanent summary line (`Write-SpectreHost`, dotted leader + `[  NNms]` total elapsed). Nested substeps leave no scrollback output; a throwing step writes no summary (state is still restored in `finally`).
- The step icon value carries **no trailing space** (e.g. `:nut_and_bolt:`, not `:nut_and_bolt: `). The single separator between the glyph and the step text is added at every render site (the spinner label, the summary line, and the breadcrumb) via `Private/Rendering/Get-StepIconPrefix.ps1`, which `TrimEnd`s the icon and appends one space (and returns `''` for an empty icon, so no stray leading space). It's idempotent, so a legacy `:nut_and_bolt: ` baked into someone's `$PROFILE` still renders with one space. When adding a render path that prints the icon, go through `Get-StepIconPrefix` rather than concatenating the raw value.
- The scriptblock handed to `Invoke-SpectreCommandWithStatus` must stay a plain literal — `.GetNewClosure()` rebinds `$script:` writes to a throwaway dynamic module and silently breaks the context stash — and its result must be swallowed (`$null =`) because the status helper emits `$null` even for a void scriptblock.
- If PwshSpectreConsole is unavailable (`Get-Command` guard), the step body still runs, silently and unrendered — failure tolerance, not a renderer.
- Anything that writes to the console mid-step (stray `Write-Host` or external-command stdout inside a step body) tears the live spinner — route output through nested `Invoke-Step` calls or suppress it (`*> $null`). `Write-Warning` stays the failure-reporting channel and is handled specially: the top-level dispatcher redirects the body's warning stream (`3>&1`, filtered to `[WarningRecord]`) into the module-scoped `$script:StepWarnings` list instead of letting it paint into the live spinner, then **replays** the captured warnings via `Write-Warning` in a `finally` after the spinner has cleared and the summary line is written. So warnings no longer tear the render *and* no longer vanish — they persist in scrollback, grouped under the top-level step's summary line (even if the step threw). A warning surfacing *after* its summary line, rather than inline, is by design.

**`Enable-*` pattern** (Enable-OhMyPosh, Enable-Zoxide, Enable-Fzf, Enable-FastNodeManager, Enable-Xh, Enable-Jq, Enable-Bat, Enable-Fd, Enable-Less): each tool enabler runs two nested substeps, `Invoke-Step "Install"` and `Invoke-Step "Initialize"`, so they render as indented timed lines under the caller's step. Some enablers self-register their own tab completion *inline* in their Initialize substep (Enable-Xh via `xh --generate complete-powershell`, Enable-Bat via `bat --completion ps1`, Enable-Fd via `fd --gen-completions powershell`) — that completer is theirs and is gated by the tool's own `-Skip` token, distinct from the external-CLI **Completion enablers** below. Where an enabler also *aliases* its tool over a builtin, it extends the generated completer's `-CommandName` to cover the alias too (PowerShell completers don't follow aliases): Enable-Xh always adds `http`/`https` (to `xh`/`xhs`), and Enable-Bat adds `cat` only under `-ReplaceCat` (the alias's completion is meaningless when `cat` is still `Get-Content`). Each `-replace` is coupled to the tool's exact `-CommandName 'xh'`/`'xhs'`/`'bat'` literal — a re-quoting in a future build silently drops only the alias's completion. (Enable-Jq and Enable-Less are install-mostly: jq has no Initialize work at all, and less only sets `$env:LESS` (and, under `-ReplaceMore`, `$env:PAGER` + a `more`→`less` alias) — GNU less ships no PowerShell completer and has no fzf/fd-style palette, so it registers no completion and isn't themed.) (Enable-Fzf is the exception that needs more than self-completion: fzf ships no PowerShell key bindings, so its Initialize installs the **PSFzf** module via `Import-ModuleSafe` and calls `Set-PsFzfOption` to bind Ctrl+T/Ctrl+R, enable fd traversal, and register the Ctrl+G git chords — see the orchestrator notes below.)
- Install: the per-tool `Invoke-Step "Install"` body is a single call to the shared `Private/Tools/Install-WingetPackageSafe.ps1` helper (the install logic is identical across the tool enablers, so it lives in one place — pass `-Id`, `-Exe`, `-CallerName`, plus optional `-PathDir` and `-Scope` (`user`/`machine`)). `-PathDir` defaults (inside the body, *after* the already-installed short-circuit, so a present tool never builds it) to the winget portable Links dir `$env:LOCALAPPDATA\Microsoft\WinGet\Links`, which every portable enabler (zoxide/fnm/fzf/xh/jq/bat/fd/less) shares — only the installer package oh-my-posh passes an explicit `-PathDir` (`Programs\oh-my-posh\bin`) and `-Scope user`. That helper installs via the first-party **`Microsoft.WinGet.Client`** module (`Install-WinGetPackage -Id <id> -Source winget -MatchOption Equals -Mode Silent [-Scope User|System]`), loaded on demand through `Import-ModuleSafe`. Crucially the already-on-PATH short-circuit (`Get-Command <Exe>`) runs **before** the module import, so an already-installed tool never loads `Microsoft.WinGet.Client` — only a first-time/missing install pays that cost (the reason package installs were moved off the startup-only `winget.exe` shell-out is acceptable). The `Install-WinGetPackage` call is wrapped in `$ProgressPreference = 'SilentlyContinue'` (restored in `finally`) so its progress bar doesn't tear `Invoke-Step`'s live Spectre spinner. After installing it patches the *current session's* `$env:Path` with the resolved `-PathDir` so the install is usable immediately (winget — CLI or cmdlet — only updates the registry user PATH, not the live process). Finally it re-checks `Get-Command <Exe>`; if the exe still isn't resolvable, `Write-Warning` surfacing the result `.Status`/`.InstallerErrorCode`. The post-install `Get-Command` re-check is the success signal — not the result code, which can be benign non-zero. The `($env:Path -split ';') -notcontains $PathDir` guard before appending PATH is deliberate — without it `$env:Path` grows on every profile reload (split-and-compare, not `-like`, so `[`/`*` in a path aren't treated as wildcards).
- Initialize: guarded by `Get-Command`, so a failed install degrades (after the Install step's warning) and startup continues.
- Initialize must run the tool's init/completion output through `Invoke-InGlobalScope (... | Out-String)`, never a bare `Invoke-Expression`. A module function that `Invoke-Expression`s the init text tags every function/alias it defines with the module's name — even `function global:` ones — so they leak into `Get-Command -Module ScrewCitySoftware.PwshProfile`. `Invoke-InGlobalScope` executes the script against the engine's top-level session state, so the helpers land in the true global scope, unattributed, exactly as if run at the prompt.
- New tool enablers should follow this exact shape. Ordering can matter: `Enable-FastNodeManager` wraps zoxide's `__zoxide_cd`, so it must be called after `Enable-Zoxide`.

**Completion enablers** — one public `Enable-<Tool>Completion` per CLI, all living under `Public/Tools/Completions/` (currently `Enable-WingetCompletion`, `Enable-AzureCliCompletion`, `Enable-TailscaleCompletion`, `Enable-DockerCompletion`, `Enable-1PasswordCompletion`, `Enable-GithubCliCompletion`). Unlike the `Enable-*` tool enablers, these only *register* tab completion — no Install/Initialize substeps — and open no `Invoke-Step` of their own; the orchestrator's Completions step (nested under **Tools**) supplies the label (e.g. `Invoke-Step "Tailscale Completions" { Enable-TailscaleCompletion }`). Each guards on its tool (`Get-Command`) so a missing CLI is a silent no-op. The registration mechanism varies by tool: winget uses a native `winget complete` argument completer; `az` is a Python/argcomplete CLI driven via temp-file + `_ARGCOMPLETE`/`COMP_*` env vars (there is **no** `az completion powershell`); `tailscale`, `op`, and `gh` are Cobra CLIs that share the **module-private** `Private/Tools/Completions/Register-CobraCompletion.ps1` engine (generates `<cmd> completion powershell` and activates it via `Invoke-InGlobalScope`, run in global scope so its helper functions/filters stay reachable at tab time and aren't tagged to the module). `tailscale`/`op` take the shell positionally; `gh` is the exception — it requires the flag form `gh completion -s powershell` (a bare positional `powershell` makes gh emit **bash**), so `Enable-GithubCliCompletion` overrides the engine's default via `Register-CobraCompletion`'s `-CompletionArgument` parameter (defaults to `@('completion','powershell')`); Docker has no built-in completion so `Enable-DockerCompletion` imports the community `DockerCompletion` module via `Import-ModuleSafe`. New completions follow this shape — a public `Enable-<Tool>Completion` in `Public/Tools/Completions/`, keeping the gnarly per-tool registration in its own function (reusing `Register-CobraCompletion` for Cobra CLIs), never inline in the orchestrator.

**Orchestrator skip contract** (`Initialize-PwshProfile`): startup is the banner plus three top-level sections — **Shell**, **Prompt**, **Tools** — and two skip parameters. `-Skip` opts out of individual tools (`Banner`, `PSReadLine`, `TerminalIcons`, `PoshGit`, `Zoxide`, `Fzf`, `Fnm`, `Xh`, `Jq`, `Bat`, `Fd`, `Less`, `Completions`); `-SkipSection` opts out of whole sections (`Shell`, `Prompt`, `Tools`). The shell completions (winget/Azure CLI/Tailscale/Docker/1Password/GitHub CLI — registration only) run as the **final nested sub-step of Tools** (completions are operations on the tools), skipped individually via `-Skip Completions` and dropped wholesale by `-SkipSection Tools` — there is no longer a Completions *section* or `-SkipSection Completions` value. The **Prompt block has no section guard — it always runs**, because oh-my-posh is table stakes and has no token in either parameter. `-SkipSection Prompt` is accepted but only drops the cosmetic extras (Terminal-Icons + posh-git) and `Write-Warning`s that oh-my-posh was kept. Keep oh-my-posh unskippable when editing this function. The theme is chosen via two mutually-exclusive parameter sets — `Bundled` (`-Theme <name>`, default `screwcity`, resolved through `Get-BundledThemePath`) and `Custom` (`-CustomTheme <path>`) — both forwarded to `Enable-OhMyPosh -Configuration`. Banner defaults: `-BannerText` defaults to `$env:COMPUTERNAME` (the live machine name) for **every** theme — it is *not* theme-specific — while `-BannerColor`/`-StepIcon`/`-FdColors`/`-FzfColors` are empty-string sentinels resolved in the body from the selected theme's branding, so `-Theme forestcity` alone yields the Forest City color/icon/fd-colors/fzf-colors (with the machine-name banner text). In the `Custom` set the body resolves color/icon from the screwcity defaults (runtime is intentionally not coupled to the custom theme). An empty/whitespace `-BannerText` renders **no** banner — the banner block is guarded on non-empty text. `bat` (`Enable-Bat`) is themed the same way: `-BatTheme` is an empty-string sentinel resolved in the body from the selected theme's branding (`Dracula` for screwcity, `gruvbox-dark` for forestcity), `-BatStyle` defaults to `numbers,changes,header`, and `-ReplaceCat` (a switch, forwarded as `-ReplaceCat:$ReplaceCat`) aliases `cat`→`bat`; the wizard prompts the cat-override inside the Features step when `bat` stays enabled (default **Yes**) and `Build-PwshProfileInitializeCall` emits it as a bare `-ReplaceCat`. `less` (`Enable-Less`) follows the same `-ReplaceCat` shape with `-ReplaceMore` (a switch, forwarded as `-ReplaceMore:$ReplaceMore`): it sets `$env:PAGER='less'` and aliases `more`→`less` so PowerShell's `help` (which honors `$env:PAGER`, else `more.com`), bat, git, delta, and gh page through less; the wizard prompts the pager-override inside the Features step when `less` stays enabled (default **Yes**) and `Build-PwshProfileInitializeCall` emits it as a bare `-ReplaceMore`. Unlike bat, less is **not** themed — it ships no PowerShell completer and has no palette, so its only always-on knob is `$env:LESS` (default `-R -F -i`, a runtime-only `-Options` param not threaded through the wizard). less is enabled after `fd`, but has no init-time ordering constraint. `fd` (`Enable-Fd`) and `fzf` (`Enable-Fzf`) are themed the same way: `-FdColors` (→ `Enable-Fd -LsColors`, sets `$env:LS_COLORS`) and `-FzfColors` (→ `Enable-Fzf -Colors`, fzf's `--color`) are empty-string sentinels resolved from branding. The fzf step is fixed-policy rather than themed for the rest: it always passes `-Style 'full'`, `-Height '100%'`, `-ProviderChord 'Ctrl+t'`/`-HistoryChord 'Ctrl+r'` (PSFzf key bindings — fzf ships none), `-GitKeyBindings`, a bat `-PreviewCommand` when `bat` is in play (scoped to `$env:FZF_CTRL_T_OPTS`, the Ctrl+T widget, so it never leaks into zoxide's directory picker), and `-UseFd:($Skip -notcontains 'Fd')`. `-Height '100%'` is written to `$env:_PSFZF_FZF_DEFAULT_OPTS` (PSFzf's widget-only opts, read in preference to `$env:FZF_DEFAULT_OPTS`): PSFzf's PSReadLine widgets otherwise force an inline `--height=40%`, so supplying a `--height` both suppresses that default and makes the Ctrl+T/Ctrl+R/git pickers fill the shell, while `$env:FZF_DEFAULT_OPTS` stays height-free so a bare `fzf` and zoxide's `cdi` keep their alternate-screen fullscreen. `Enable-Fd` is enabled **after** `Enable-Fzf` so `fzf.exe` is on PATH for its `-IntegrateFzf` check; none of the fd/fzf knobs are wizard-customizable, so they're not threaded through `Get-PwshProfileDefault`/`Build-PwshProfileInitializeCall`. The install **wizard**, by contrast, seeds a custom theme's color/icon prompts with **neutral** defaults (`Silver`, `:gear:`) rather than the screwcity identity, so the user brands a custom theme fresh (the banner text still defaults to `$env:COMPUTERNAME`); those choices are emitted explicitly into the generated `Initialize-PwshProfile` call.

**PwshSpectreConsole output gotcha**: `Write-SpectreFigletText` and the `Format-*` widgets emit their rendered ANSI string to the *pipeline* (not the console) — at a prompt the formatter prints it, but anything invoked from the `.psm1` at import time must pipe to `Out-Host` or the output is silently discarded. `Write-SpectreHost` and the live displays write to the console directly and don't need this.

**Install-wizard chrome & help-markup convention**: the `Install-PwshProfile` wizard's look is built from three `Private/Install/` helpers. `Format-PwshProfileHelpMarkup` is the **single source of truth for highlighting** — it converts a markdown-ish convention to safe Spectre markup: `**term**` → accent (`#c9aaff`, tool/product names), `` `code` `` → cyan (`#5fd7ff`, file types/commands/paths), everything else escaped (via `Get-SpectreEscapedText`) and wrapped in a body style (`grey` by default; pass `-Body default` for no wrapper). **Write help strings in this convention** anywhere they flow through `Write-PwshProfilePromptHelp` (inline `›` hint lines) or `Write-PwshProfileStepHeader` (the per-step rounded panel: `"◆ Title · N of M"` header — plain text, color from the border, Spectre trims any padding spaces — over a formatted body). Both `Format-*`/panel helpers emit to the pipeline, so `Write-PwshProfileStepHeader` pipes its panel to `Out-Host` internally (the leak the bare step calls would otherwise expose; there's a test for it). The wizard threads each step's 1-based index + total into the header; `$state` carries the fixed UI colors `Accent`/`Code`, intentionally decoupled from the prompt theme being configured. Keep the wizard's `-Message` strings stable — the wizard tests key mocks on them. `Read-SpectreSelection` wraps Spectre's interactive `SelectionPrompt`, which **clears itself on submit** (no persist toggle), so the chosen value would vanish — unlike `Read-SpectreText`, which leaves its answer on screen; the wizard therefore echoes each selection (Theme/alignment/font/step icon) via `Write-PwshProfilePromptAnswer` (accent `✓` + grey value, escape-only). Don't advertise "leave blank" escape hatches for prompts that carry a pre-filled `-DefaultAnswer` (Enter keeps the default, not blank).

**Bundled assets live under `Assets/`** (`Assets/Themes/` for the oh-my-posh themes, `Assets/Fonts/` for FIGlet `.flf` fonts). Paths are resolved through `Private/` helpers so there's one source of truth each: `Get-BundledThemePath -Name <theme>` (in `Private/Prompt/` → `Assets/Themes/<theme>.omp.json`, default `screwcity`) and `Get-BundledFontPath -Name <font>` (in `Private/Rendering/` → `Assets/Fonts/<font>.flf`). `$PSScriptRoot` in those helpers is `Private/<area>/` (two levels deep), so they `Join-Path '..' '..' 'Assets' ...` to reach the module root — the same applies to `Show-PwshProfileReadme` in `Public/Docs/` resolving the README. The banner is rendered by the generic `Write-Figlet` (formerly `Show-ProfileBanner`): a pure figlet writer with mutually-exclusive `-Font` / `-FontPath` parameter sets, no trailing blank line, defaults of `-Font ANSIShadow` and `-Color '#c9aaff'` (the screwcity theme's signature purple), and a missing-bundled-font fallback to the default font.

- **Two bundled themes, palette-only twins.** `forestcity.omp.json` is a copy of `screwcity.omp.json` with **only** the `palette` block recolored (every palette *key* name preserved, so all segment references resolve unchanged) — when editing one theme's structure, keep the other in sync, and recolor by palette values, never by rewriting segments. Validate a theme change parses as JSON and (ideally) renders. Each palette intentionally carries a **full color ramp**, so roughly half its keys are referenced by no segment — that's deliberate (it keeps the two palettes symmetric and gives a ready customization surface); don't "clean up" the unreferenced keys. The set of bundled themes is **discovered dynamically** from `Assets/Themes/*.omp.json` via `Private/Prompt/Get-BundledThemeName.ps1` (strips the `.omp.json` suffix) — there is **no hardcoded theme list**. `Initialize-PwshProfile -Theme` (parameter set `Bundled`, mutually exclusive with `-CustomTheme` in set `Custom`), `Get-OhMyPoshTheme -Theme`, and `Export-OhMyPoshTheme -Theme` validate with `[ValidateScript({ $_ -in (Get-BundledThemeName) })]` (module scope, so the private helper is reachable) and tab-complete via an `[ArgumentCompleter]` that globs `(Get-Module ScrewCitySoftware.PwshProfile).ModuleBase` (completers run in the *caller's* scope, where module-private functions are NOT visible). Each theme's **branding** — `DisplayName` (the friendly label for the wizard's theme picker), banner color, step icon, `BatTheme` (the `bat --list-themes` value driving `$env:BAT_THEME`), `LsColors` (the truecolor `LS_COLORS` spec `Enable-Fd` assigns to `$env:LS_COLORS`), and `FzfColors` (the fzf `--color` spec folded into `$env:FZF_DEFAULT_OPTS`) — lives in `Private/Prompt/Get-BundledThemeBranding.ps1`, a small explicit two-entry map with screwcity as the fallback for any unknown/custom theme. (Banner *text* is **not** here: it defaults uniformly to `$env:COMPUTERNAME`.) Both `Initialize-PwshProfile` (fills banner color/icon/bat-theme/fd-colors/fzf-colors not explicitly passed) and `Get-PwshProfileDefault -Theme` (seeds the wizard + the `Build-PwshProfileInitializeCall` comparison baseline) resolve those through it, so adding a third bundled theme means dropping a `.omp.json` **and** (if it wants its own color/icon/bat/fd/fzf identity) adding a branding entry. (`Get-PwshProfileDefault` carries only the wizard-customizable subset — banner color/icon, bat theme — not fd/fzf colors, which are runtime-only.)

- **The accepted `-Font` set is discovered dynamically** from `Assets/Fonts/*.flf` via `Private/Rendering/Get-BundledFontName.ps1` — there is **no hardcoded font list**. `Write-Figlet`, `Show-FigletFont`, and `Initialize-PwshProfile -BannerFont` each validate `-Font` with `[ValidateScript({ $_ -in (Get-BundledFontName) })]` (runs in module scope, so the private helper is reachable) and tab-complete via an `[ArgumentCompleter]` that calls the **public** `Show-FigletFont` (completers run in the *caller's* scope, where module-private functions are NOT visible — calling the private helper there silently falls back to file completion). `Show-FigletFont` lists the names by default and renders samples under `-Preview`.
- Adding/removing a font is just dropping/deleting a `.flf` in `Assets/Fonts/` (file base name == the `-Font` value, no spaces). Not every `.flf` works under Spectre's parser: some throw on load (`Big.flf`/`Digital`/`Georgia11`) and some load but render **blank** (`Roman`). **Render-test any new font for non-empty output** — `(Write-SpectreFigletText 'Screw City' -FigletFontPath <f> | Out-String)` should have non-whitespace chars (an exception-only check is insufficient) — before committing, and record source/license in `Assets/Fonts/README.md`.

**Failure tolerance is a design rule**: nothing in this module may throw out of profile startup. Failures become `Write-Warning` + early return (see `Public/Core/Import-ModuleSafe.ps1`) or are silently skipped behind a `Get-Command` guard.
